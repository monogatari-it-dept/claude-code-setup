チーム開発ガイドライン docs/TEAM_DEVELOPMENT_GUIDE.md

Anthropic公式ベストプラクティスに基づくチーム開発運用ガイド

1. 基本原則

1.1 検証可能な作業を心がける

Claude Codeの品質を最大化する最も重要なポイントは「Claudeが自分の作業を検証できる手段を提供する」こと。

  • テストコードを書いてから実装を依頼する（TDD）
  • 期待する出力やスクリーンショットを提示する
  • npm test や npm run lint など検証コマンドを明示する
  • 根本原因の解決を求め、エラー抑制は避ける

1.2 探索 → 計画 → 実装 → コミット

  1. 探索（Plan Mode）: コードベースを読み、構造を理解する
  2. 計画（Plan Mode）: 実装計画を作成し、レビューする
  3. 実装（Normal Mode）: 計画に沿ってコードを書き、テストで検証する
  4. コミット: 説明的なメッセージでコミットし、PRを作成する

小さな修正（typo、ログ追加、変数名変更）は計画をスキップして直接実装してよい。

1.3 具体的なコンテキストを提供する

悪い例良い例
「テストを追加して」「foo.pyにログアウト時のエッジケースのテストを書いて。モックは避けて」
「ログインバグを直して」「セッションタイムアウト後にログインが失敗する。src/auth/のトークンリフレッシュを確認して、再現テストを書いてから修正して」
「カレンダーウィジェットを追加して」「HotDogWidget.phpのパターンを参考に、月選択とページネーション付きのカレンダーウィジェットを実装して」

2. コンテキスト管理

2.1 コンテキストウィンドウは最重要リソース

Claudeのコンテキストウィンドウは会話全体、読み込んだファイル、コマンド出力で埋まる。パフォーマンスはコンテキストが埋まるほど低下する。

  • 無関係なタスク間では /clear でリセットする
  • 2回以上の修正で解決しない場合は /clear して最初からやり直す
  • 調査タスクはサブエージェントに委譲してメインコンテキストを保護する
  • /compact で会話を要約してコンテキストを節約する

2.2 セッション管理

  • claude --continue: 直前のセッションを再開
  • claude --resume: 過去のセッション一覧から選択
  • /rename: セッションに名前をつけて管理（例: oauth-migration）

4. CLAUDE.md の書き方

4.1 基本方針

  • 200行以内に収める（長すぎるとClaudeが指示を見落とす）
  • Claudeが推測できない情報だけを書く
  • 具体的で検証可能な指示にする
  • 定期的にレビューし、不要な項目を削除する

4.2 書くべき内容

  • Claudeが推測できないBashコマンド
  • デフォルトと異なるコードスタイルルール
  • テスト実行方法、推奨テストランナー
  • ブランチ命名規則、PR規約
  • プロジェクト固有のアーキテクチャ決定
  • 環境変数や開発環境の特殊設定
  • よくあるハマりポイント

4.3 書くべきでない内容

  • Claudeがコードを読めばわかること
  • 言語の標準的な規約
  • 詳細なAPIドキュメント（リンクで代替）
  • 頻繁に変更される情報
  • ファイルごとの説明
  • 「クリーンなコードを書く」のような自明な指示

5. チーム開発フロー

5.1 ブランチ戦略

main ← develop ← feature/xxx
                ← fix/xxx
                ← refactor/xxx

5.2 コミットメッセージ規約

<type>(<scope>): <description>

feat:     新機能
fix:      バグ修正
refactor: リファクタリング
test:     テスト追加・修正
docs:     ドキュメント
chore:    ビルド・設定変更

5.3 PRフロー

  1. featureブランチで開発
  2. Claude Codeでテスト実行・リント通過を確認
  3. PRを作成（テンプレートに従う）
  4. 別セッションのClaudeでレビュー（Writer/Reviewerパターン）
  5. レビュー指摘を修正
  6. マージ

5.4 Writer/Reviewer パターン

セッションA（Writer）セッションB（Reviewer）
機能を実装—
—PRのレビューを実施。エッジケース、レースコンディション、既存パターンとの一貫性を確認
レビューフィードバックを元に修正—

6. 禁止事項・注意事項

6.1 よくあるアンチパターン

アンチパターン対策
キッチンシンクセッションタスク間で /clear する
何度も修正を繰り返す2回失敗したら /clear して良いプロンプトで再開
CLAUDE.mdが肥大化不要な指示を削除。コードを読めばわかることは書かない
検証なしで信頼テスト、スクリプト、スクリーンショットで必ず検証
無限探索調査はスコープを絞るか、サブエージェントに委譲

6.2 セキュリティ

  • 機密情報（APIキー、パスワード）をCLAUDE.mdやコミットに含めない
  • .env ファイルは .gitignore に含める
  • セキュリティレビュー用サブエージェントを活用する

7. スキル・フック・サブエージェントの使い分け

機能用途実行タイミング
CLAUDE.md常に適用するルール・規約毎セッション開始時に自動読み込み
.claude/rules/パス別ルール対象ファイル操作時に自動読み込み
Skills再利用可能なワークフローユーザー or Claude が必要時に起動
Hooks確実に実行すべき自動処理特定イベント発生時に必ず実行
Subagents隔離された専門タスクClaude が委譲 or ユーザー指示時

GitHub CLIのインストールからリポジトリ作成、チームメンバーの初期設定まで

1. GitHub CLIのインストール

macOS（Homebrew）

brew install gh

Windows

winget install --id GitHub.cli

Linux（Ubuntu/Debian）

(type -p wget >/dev/null || (sudo apt update && sudo apt-get install wget -y)) \
  && sudo mkdir -p -m 755 /etc/apt/keyrings \
  && out=$(wget -nv -O- https://cli.github.com/packages/githubcli-archive-keyring.gpg) \
  && echo "$out" | sudo tee /etc/apt/keyrings/githubcli-archive-keyring.gpg > /dev/null \
  && sudo chmod go+r /etc/apt/keyrings/githubcli-archive-keyring.gpg \
  && echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/githubcli-archive-keyring.gpg] https://cli.github.com/packages stable main" | sudo tee /etc/apt/sources.list.d/github-cli.list > /dev/null \
  && sudo apt update \
  && sudo apt install gh -y

2. GitHub CLIの認証

gh auth login

対話的に以下を選択:

  1. GitHub.com を選択
  2. HTTPS を選択
  3. 認証方法: Login with a web browser を選択
  4. 表示されるワンタイムコードをブラウザで入力

認証の確認:

gh auth status

3. リポジトリの作成とプッシュ

# テンプレートフォルダに移動
cd ~/Documents/claude-code-team-template

# GitHubにプライベートリポジトリを作成してプッシュ

VS Code + Claude Code 開発環境 docs/VSCODE_SETUP_GUIDE.md

チーム標準エディタとしてVS Codeを推奨し、Claude Code拡張機能を統合した開発環境を構築する

なぜ VS Code + Claude Code か

VS Codeは軽量で拡張性が高く、チーム全体で統一した開発体験を実現できる。Claude Code拡張機能を統合することで、エディタから離れることなくAIアシスタントとの対話、コードレビュー、自動修正が可能になる。

  • エディタ内のパネルで直接Claudeと対話できる
  • ファイルの@メンション（行番号指定も可能）でピンポイントな質問ができる
  • 変更のインラインdiff表示で、提案内容を確認してから適用できる
  • 複数の会話を並行して開ける（タブ/ウィンドウ分割）
  • Plan Modeで実装計画をレビューしてから作業を開始できる
  • CLAUDE.md、Skills、Hooks、Rulesがそのまま動作する

スキルの使い方（VS Code内）

プロンプトボックスで / を入力するとコマンドメニューが開く:

/fix-issue 42        # Issue #42 を修正
/review-pr 15        # PR #15 をレビュー
/commit              # 変更をコミット
/create-api users    # users APIを生成

チーム開発ベストプラクティス

Writer/Reviewer パターン（VS Code版）

  1. Writerセッション: メインのClaudeパネルで機能を実装
  2. Reviewerセッション: 「Open in New Tab」で新しいClaudeタブを開き、PRレビューを依頼（/review-pr 15）

Plan Mode活用

  1. プロンプトボックス下部のモード表示をクリック → Plan を選択
  2. 実装したい機能を説明する
  3. VS Codeが自動的に計画をMarkdownドキュメントとして表示
  4. 計画にインラインコメントでフィードバックを追加
  5. 承認後にNormal Modeに戻して実装を開始

チェックポイント（Undo機能）

Claudeのメッセージにホバーすると巻き戻しボタンが表示される:

  • Fork conversation from here: この地点から会話を分岐（コードは変更なし）
  • Rewind code to here: この地点にコードを巻き戻し（会話履歴は保持）
  • Fork and rewind: 会話を分岐しつつコードも巻き戻し

Git Worktreeで並行開発

claude --worktree feature-auth       # 認証機能の開発
claude --worktree feature-dashboard  # ダッシュボードの開発

各worktreeは独立したファイル状態を保持しながらGit履歴を共有する。Claudeインスタンス間のファイル競合を防げる。

  • IMPORTANT: コード変更後は必ず型チェック（npx tsc --noEmit）を実行すること
  • テスト実行時はパフォーマンスのため単一テストを推奨。全体テストはコミット前のみ
  • コミットメッセージは <type>(<scope>): <description> 形式
  • ブランチ名は feature/xxx、fix/xxx、refactor/xxx 形式

  • .env ファイルは絶対にコミットしない

  • データベースマイグレーションファイルは手動で確認する
  • 外部API呼び出しには必ずエラーハンドリングとリトライを実装する

フロントエンドルール .claude/rules/frontend.md

対象パス: src/client/**/*.{ts,tsx}, src/client/**/*.css
  • Reactコンポーネントは関数コンポーネントで作成する（クラスコンポーネントは禁止）
  • 状態管理にはReact Hooks（useState, useReducer, useContext）を使用する
  • コンポーネントファイル名は PascalCase（例: UserProfile.tsx）
  • CSSモジュールまたはTailwind CSSを使用する。インラインスタイルは避ける
  • ページコンポーネントは src/client/pages/ に配置
  • 共通コンポーネントは src/client/components/ に配置
  • カスタムHooksは src/client/hooks/ に配置
  • APIコールは src/client/api/ に集約する（コンポーネント内で直接fetchしない）
  • すべてのユーザー入力にバリデーションを実装する
  • エラーバウンダリを適切に配置する

バックエンドルール .claude/rules/backend.md

対象パス: src/server/**/*.ts
  • APIエンドポイントはRESTful設計に従う
  • URLパスは kebab-case、JSONプロパティは camelCase
  • リストエンドポイントには必ずページネーションを実装する
  • すべてのエンドポイントに入力バリデーションを実装する（zod を使用）
  • エラーレスポンスは統一フォーマット: { error: { code: string, message: string } }
  • データベースアクセスは src/server/repositories/ に集約
  • ビジネスロジックは src/server/services/ に配置
  • ミドルウェアは src/server/middleware/ に配置
  • 外部API呼び出しにはリトライとタイムアウトを必ず設定する
  • 環境変数は src/server/config.ts で一元管理（process.envを直接参照しない）
  • ログ出力はロガーライブラリ経由で行う（console.log は禁止）

テストルール .claude/rules/testing.md

対象パス: **/*.test.ts, **/*.test.tsx, **/*.spec.ts
  • テストファイルはソースファイルと同階層に .test.ts で作成する
  • テスト名は日本語で記述してよい（可読性を重視）
  • Arrange-Act-Assert パターンに従う
  • モックは最小限にする。実際のデータベースやAPIに依存する場合のみ使用
  • テストデータはファクトリ関数で作成する（ハードコーディングしない）
  • 各テストは他のテストに依存しない独立したものにする
  • エッジケース（null, undefined, 空配列, 境界値）を必ずテストする
  • 非同期処理のテストには async/await を使用する
  • カバレッジ目標: ビジネスロジック80%以上、ユーティリティ関数90%以上

fix-issue .claude/skills/fix-issue/SKILL.md

GitHub Issueを分析して修正するワークフロー

disable-model-invocation: true — ユーザーが /fix-issue で手動起動
allowed-tools: Bash(gh *) Bash(git *) Bash(npm test *) Bash(npx tsc *)

使い方: /fix-issue 42（Issue番号を指定）

ワークフロー

  1. gh issue view $ARGUMENTS でIssueの詳細を取得
  2. Issue内容を分析し、関連するコードを特定
  3. 修正用ブランチを作成: git checkout -b fix/$ARGUMENTS-<簡潔な説明>
  4. 問題を再現するテストを作成する（テストが失敗することを確認）
  5. 修正を実装する
  6. テストが通ることを確認: npm test
  7. 型チェック: npx tsc --noEmit
  8. リント: npm run lint
  9. 説明的なコミットメッセージで変更をコミット
  10. PRを作成: gh pr create

test-writer .claude/agents/test-writer.md

ソースコードを分析してテストケースを生成する専門エージェント

tools: Read, Grep, Glob, Write, Bash
model: sonnet

テスト作成方針

  1. 対象コードを読んで、関数やクラスの責務を理解する
  2. 既存のテストパターンを確認する
  3. テストケースを設計する:
    • 正常系 期待通りの入力での動作
    • 異常系 不正な入力、エラー条件での動作

  4. Arrange-Act-Assert パターンでテストコードを作成
  5. テスト実行して全テストがパスすることを確認する
