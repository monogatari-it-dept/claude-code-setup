はじめてのプロジェクト

環境セットアップ後に、Claude Codeを使って簡単なウェブページを作成し、GitHub Pagesで公開するまでの一連の流れを体験します。所要時間: 約15分

このチュートリアルでは、新しいリポジトリの作成からClaude Code設定、Hello Worldページの実装、GitHub Pagesでの公開、プロジェクト一覧での確認までを一気通貫で行います。セットアップガイドが完了していることが前提です。

0. 前提条件の確認

以下がすべて完了していることを確認してください:

項目確認コマンド期待結果
Claude Code CLIclaude --versionバージョン番号が表示される
GitHub CLIgh auth statusLogged in と表示される
VS Code + 拡張機能code --list-extensions | grep anthropicanthropic.claude-code
Git設定git config user.name && git config user.email名前とメールが表示される
💡 code や gh コマンドが見つからない場合は、PATHの設定が必要です。VS Code は起動後に Cmd+Shift+P → 「Shell Command: Install 'code' command in PATH」を実行してください。Homebrew 未導入の場合は /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" でインストール後、eval "$(/opt/homebrew/bin/brew shellenv zsh)" を実行してから brew install gh を行ってください。

1. リポジトリを作成する

GitHub CLI を使って、組織内に新しいリポジトリを作成します。

# リポジトリ名は自由に変更してください
gh repo create monogatari-it-dept/hello-world \
  --public \
  --description "My first Claude Code project" \
  --clone

cd hello-world
💡 --public は GitHub Pages（Freeプラン）で公開するために必要です。

2. Claude Code の設定ファイルを配置する

チームテンプレートから設定ファイルをコピーするか、手動で作成します。

# ディレクトリ構造を作成
mkdir -p .claude/rules .claude/skills .vscode docs

# CLAUDE.md を作成（プロジェクトの基本ルール）
cat <<'EOF' > CLAUDE.md
# Hello World Project

## Build & Run
- Open docs/index.html in browser

## Code Style
- HTML5 semantic tags
- CSS: mobile-first, no frameworks
- JavaScript: vanilla ES6+, no dependencies

## Architecture
- Single page application in docs/
- GitHub Pages deployment from /docs on main branch
EOF

# チーム共通の settings.json をコピー
cat <<'EOF' > .claude/settings.json
{
  "permissions": {
    "allow": [
      "git status", "git log", "git diff",
      "git add", "git commit", "git push", "git pull"
    ],
    "deny": [
      "git push --force",
      "git reset --hard",
      "rm -rf /"
    ]
  }
}
EOF

# VS Code 設定をコピー
cat <<'EOF' > .vscode/settings.json
{
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.formatOnSave": true,
  "editor.tabSize": 2
}
EOF

# .gitignore
cat <<'EOF' > .gitignore
node_modules/
.env
CLAUDE.local.md
.claude/settings.local.json
.DS_Store
EOF

3. Hello World ページを作成する

方法A: Claude Code に依頼する（推奨）

VS Code で Claude パネルを開き（Sparkアイコン ✱）、以下のように依頼します:

Claude への指示例:
docs/index.html に Hello World ウェブページを作って。 要件: - ダークテーマ - ページ中央に「Hello World from monogatari-it-dept!」と表示 - 現在日時をリアルタイムで表示する時計機能 - CSS/JSはHTMLファイル内にインライン記述 - レスポンシブ対応
💡 生成されたコードに修正を加えたい場合は、続けて「時計のフォントをもう少し大きくして」「背景にグラデーションを追加して」のように自然言語で指示できます。

方法B: 手動で作成する

ターミナルで以下を実行すると、動作するサンプルページが作成されます:

cat <<'EOF' > docs/index.html

    body {

ブラウザで直接ファイルを開いて確認します:

# Windows

# Linux

5. コミットしてプッシュする

方法A: Claude Code に依頼する

Claude への指示例:
変更をすべてコミットしてプッシュして。 コミットメッセージ: 「feat: Hello Worldページを作成」

方法B: 手動で実行する

# すべてのファイルをステージング
git add -A

# コミット
git commit -m "feat: Hello Worldページを作成"

# デフォルトブランチが master の場合は main にリネーム
# （GitHub Pages は main ブランチを参照するため必須）
git branch -m master main

# プッシュ（初回は -u でトラッキング設定）
git push -u origin main
💡 git branch -m master main は初回のみ必要です。gh repo create がデフォルトで master ブランチを作成するため、main にリネームします。2回目以降のプッシュは git push だけでOKです。

6. GitHub Pages を有効化する

GitHub CLI を使って Pages を設定します:

# GitHub Pages を有効化（/docs ディレクトリから配信）
gh api repos/monogatari-it-dept/hello-world/pages \
  -X POST \
  --input - <<'EOF'
{
  "build_type": "legacy",
  "source": {
    "branch": "main",
    "path": "/docs"
  }
}
EOF

1〜2分待つと、以下の URL でページが公開されます:

https://monogatari-it-dept.github.io/hello-world/
💡 初回のデプロイには数分かかることがあります。404が表示される場合は少し待ってからリロードしてください。

デプロイ状況の確認コマンド:

# Pages のステータス確認
gh api repos/monogatari-it-dept/hello-world/pages --jq '.status'

# 最新のビルド状況を確認
gh api repos/monogatari-it-dept/hello-world/pages/builds --jq '.[0].status'

# docs/index.html がリポジトリに存在するか確認
gh api repos/monogatari-it-dept/hello-world/contents/docs/index.html --jq '.name'

チーム開発ガイドライン docs/TEAM_DEVELOPMENT_GUIDE.md

Anthropic公式ベストプラクティスに基づくチーム開発運用ガイド

1. 基本原則

1.1 検証可能な作業を心がける

Claude Codeの品質を最大化する最も重要なポイントは「Claudeが自分の作業を検証できる手段を提供する」こと。

  • テストコードを書いてから実装を依頼する（TDD）
  • 期待する出力やスクリーンショットを提示する
  • npm test や npm run lint など検証コマンドを明示する
  • 根本原因の解決を求め、エラー抑制は避ける

1.2 探索 → 計画 → 実装 → コミット

  1. 探索（Plan Mode）: コードベースを読み、構造を理解する
  2. 計画（Plan Mode）: 実装計画を作成し、レビューする
  3. 実装（Normal Mode）: 計画に沿ってコードを書き、テストで検証する
  4. コミット: 説明的なメッセージでコミットし、PRを作成する

小さな修正（typo、ログ追加、変数名変更）は計画をスキップして直接実装してよい。

1.3 具体的なコンテキストを提供する

悪い例良い例
「テストを追加して」「foo.pyにログアウト時のエッジケースのテストを書いて。モックは避けて」
「ログインバグを直して」「セッションタイムアウト後にログインが失敗する。src/auth/のトークンリフレッシュを確認して、再現テストを書いてから修正して」
「カレンダーウィジェットを追加して」「HotDogWidget.phpのパターンを参考に、月選択とページネーション付きのカレンダーウィジェットを実装して」

2. コンテキスト管理

2.1 コンテキストウィンドウは最重要リソース

Claudeのコンテキストウィンドウは会話全体、読み込んだファイル、コマンド出力で埋まる。パフォーマンスはコンテキストが埋まるほど低下する。

  • 無関係なタスク間では /clear でリセットする
  • 2回以上の修正で解決しない場合は /clear して最初からやり直す
  • 調査タスクはサブエージェントに委譲してメインコンテキストを保護する
  • /compact で会話を要約してコンテキストを節約する

2.2 セッション管理

  • claude --continue: 直前のセッションを再開
  • claude --resume: 過去のセッション一覧から選択
  • /rename: セッションに名前をつけて管理（例: oauth-migration）

4. CLAUDE.md の書き方

4.1 基本方針

  • 200行以内に収める（長すぎるとClaudeが指示を見落とす）
  • Claudeが推測できない情報だけを書く
  • 具体的で検証可能な指示にする
  • 定期的にレビューし、不要な項目を削除する

4.2 書くべき内容

  • Claudeが推測できないBashコマンド
  • デフォルトと異なるコードスタイルルール
  • テスト実行方法、推奨テストランナー
  • ブランチ命名規則、PR規約
  • プロジェクト固有のアーキテクチャ決定
  • 環境変数や開発環境の特殊設定
  • よくあるハマりポイント

4.3 書くべきでない内容

  • Claudeがコードを読めばわかること
  • 言語の標準的な規約
  • 詳細なAPIドキュメント（リンクで代替）
  • 頻繁に変更される情報
  • ファイルごとの説明
  • 「クリーンなコードを書く」のような自明な指示

VS Code + Claude Code 開発環境 docs/VSCODE_SETUP_GUIDE.md

チーム標準エディタとしてVS Codeを推奨し、Claude Code拡張機能を統合した開発環境を構築する

なぜ VS Code + Claude Code か

VS Codeは軽量で拡張性が高く、チーム全体で統一した開発体験を実現できる。Claude Code拡張機能を統合することで、エディタから離れることなくAIアシスタントとの対話、コードレビュー、自動修正が可能になる。

  • エディタ内のパネルで直接Claudeと対話できる
  • ファイルの@メンション（行番号指定も可能）でピンポイントな質問ができる
  • 変更のインラインdiff表示で、提案内容を確認してから適用できる
  • 複数の会話を並行して開ける（タブ/ウィンドウ分割）
  • Plan Modeで実装計画をレビューしてから作業を開始できる
  • CLAUDE.md、Skills、Hooks、Rulesがそのまま動作する

CLAUDE.md プロジェクトルート

毎セッション開始時に自動読み込みされるプロジェクト共通ルール

ビルド・実行コマンド

パッケージマネージャnpm（yarn, pnpmは不使用）

ビルドnpm run build

コードスタイル

  • ES Modules（import/export）を使用。CommonJS（require）は禁止
  • import時はdestructureを推奨
  • インデントは2スペース
  • セミコロンは使用する
  • 文字列はシングルクォート
  • 変数・関数名は camelCase、クラス名は PascalCase、定数は UPPER_SNAKE_CASE
  • 型定義は interface を優先（type は合併型や交差型に限定）
  • any 型は禁止。unknown + 型ガードを使用する

ワークフロー

  • IMPORTANT: コード変更後は必ず型チェック（npx tsc --noEmit）を実行すること
  • テスト実行時はパフォーマンスのため単一テストを推奨。全体テストはコミット前のみ
  • コミットメッセージは <type>(<scope>): <description> 形式
  • ブランチ名は feature/xxx、fix/xxx、refactor/xxx 形式

アーキテクチャ

レイヤーパス
フロントエンドsrc/client/
バックエンドsrc/server/
共有型定義src/shared/types/
APIエンドポイントsrc/server/api/（RESTful設計）
テスト各ソースファイルと同階層に .test.ts

注意事項

  • .env ファイルは絶対にコミットしない
  • package-lock.json は変更しない（npm install 以外で）
  • データベースマイグレーションファイルは手動で確認する
  • 外部API呼び出しには必ずエラーハンドリングとリトライを実装する

フロントエンドルール .claude/rules/frontend.md

対象パス: src/client/**/*.{ts,tsx}, src/client/**/*.css
  • Reactコンポーネントは関数コンポーネントで作成する（クラスコンポーネントは禁止）
  • 状態管理にはReact Hooks（useState, useReducer, useContext）を使用する
  • コンポーネントファイル名は PascalCase（例: UserProfile.tsx）
  • CSSモジュールまたはTailwind CSSを使用する。インラインスタイルは避ける
  • ページコンポーネントは src/client/pages/ に配置
  • 共通コンポーネントは src/client/components/ に配置
  • カスタムHooksは src/client/hooks/ に配置
  • APIコールは src/client/api/ に集約する（コンポーネント内で直接fetchしない）
  • すべてのユーザー入力にバリデーションを実装する
  • エラーバウンダリを適切に配置する

バックエンドルール .claude/rules/backend.md

対象パス: src/server/**/*.ts
  • APIエンドポイントはRESTful設計に従う
  • URLパスは kebab-case、JSONプロパティは camelCase
  • リストエンドポイントには必ずページネーションを実装する
  • すべてのエンドポイントに入力バリデーションを実装する（zod を使用）
  • エラーレスポンスは統一フォーマット: { error: { code: string, message: string } }
  • データベースアクセスは src/server/repositories/ に集約
  • ビジネスロジックは src/server/services/ に配置
  • ミドルウェアは src/server/middleware/ に配置
  • 外部API呼び出しにはリトライとタイムアウトを必ず設定する
  • 環境変数は src/server/config.ts で一元管理（process.envを直接参照しない）
  • ログ出力はロガーライブラリ経由で行う（console.log は禁止）

テストルール .claude/rules/testing.md

対象パス: **/*.test.ts, **/*.test.tsx, **/*.spec.ts
  • テストファイルはソースファイルと同階層に .test.ts で作成する
  • テスト名は日本語で記述してよい（可読性を重視）
  • Arrange-Act-Assert パターンに従う
  • モックは最小限にする。実際のデータベースやAPIに依存する場合のみ使用
  • テストデータはファクトリ関数で作成する（ハードコーディングしない）
  • 各テストは他のテストに依存しない独立したものにする
  • エッジケース（null, undefined, 空配列, 境界値）を必ずテストする
  • 非同期処理のテストには async/await を使用する
  • カバレッジ目標: ビジネスロジック80%以上、ユーティリティ関数90%以上

ワークフロー

  1. gh issue view $ARGUMENTS でIssueの詳細を取得
  2. Issue内容を分析し、関連するコードを特定
  3. 修正用ブランチを作成: git checkout -b fix/$ARGUMENTS-<簡潔な説明>
  4. 問題を再現するテストを作成する（テストが失敗することを確認）
  5. 修正を実装する
  6. テストが通ることを確認: npm test
  7. 型チェック: npx tsc --noEmit
  8. リント: npm run lint
  9. 説明的なコミットメッセージで変更をコミット
  10. PRを作成: gh pr create

review-pr .claude/skills/review-pr/SKILL.md

PRのコードレビューを実施する

context: fork — フォークされたサブエージェントで実行（メインコンテキストを保護）
agent: Explore
allowed-tools: Bash(gh *)

使い方: /review-pr 15（PR番号を指定）

動的コンテキスト注入

  • !`gh pr diff $ARGUMENTS` — PR差分
  • !`gh pr view $ARGUMENTS` — PR情報
  • !`gh pr diff $ARGUMENTS --name-only` — 変更ファイル一覧

レビュー観点

  1. バグ・ロジックエラー: エッジケースの見落とし、off-by-oneエラー、null/undefinedの未チェック
  2. セキュリティ: SQLインジェクション、XSS、認証・認可の不備
  3. パフォーマンス: N+1クエリ、不要な再レンダリング、メモリリーク
  4. コード品質: 命名、責務の分離、DRY原則
  5. アーキテクチャ: 既存パターンとの一貫性、依存関係の方向

項目内容
深刻度🔴 Critical / 🟡 Warning / 🔵 Info
ファイルファイルパスと行番号
問題具体的な問題の説明
提案修正案

security-reviewer .claude/agents/security-reviewer.md

コードのセキュリティ脆弱性をレビューする専門エージェント

tools: Read, Grep, Glob, Bash
model: sonnet

レビュー対象

  1. インジェクション脆弱性: SQLインジェクション、XSS、コマンドインジェクション
  2. 認証・認可の不備: 認証バイパス、権限昇格、セッション管理の問題
  3. 機密情報の露出: コード内のシークレット、ログへの機密情報出力
  4. 安全でないデータ処理: 暗号化の不備、入力の未検証、安全でないデシリアライゼーション
  5. 依存関係の脆弱性: 既知の脆弱性を持つパッケージの使用

出力形式

項目内容
深刻度Critical / High / Medium / Low
場所ファイルパスと行番号
種類脆弱性の種類
シナリオ攻撃シナリオの説明

test-writer .claude/agents/test-writer.md

ソースコードを分析してテストケースを生成する専門エージェント

tools: Read, Grep, Glob, Write, Bash
model: sonnet

テスト作成方針

  1. 対象コードを読んで、関数やクラスの責務を理解する
  2. 既存のテストパターンを確認する
  3. テストケースを設計する:
    • 正常系 期待通りの入力での動作
    • 異常系 不正な入力、エラー条件での動作
    • 境界値 null, undefined, 空配列, 最大値/最小値
    • 並行処理 非同期処理のレースコンディション
  4. Arrange-Act-Assert パターンでテストコードを作成
  5. テスト実行して全テストがパスすることを確認する
