# Claude Code チーム開発ガイドライン

> Anthropic公式ベストプラクティスに基づくチーム開発運用ガイド

## 1. 基本原則

### 1.1 検証可能な作業を心がける

Claude Codeの品質を最大化する最も重要なポイントは「Claudeが自分の作業を検証できる手段を提供する」こと。

- テストコードを書いてから実装を依頼する（TDD）
- 期待する出力やスクリーンショットを提示する
- `npm test` や `npm run lint` など検証コマンドを明示する
- 根本原因の解決を求め、エラー抑制は避ける

### 1.2 探索 → 計画 → 実装 → コミット

1. **探索（Plan Mode）**: コードベースを読み、構造を理解する
2. **計画（Plan Mode）**: 実装計画を作成し、レビューする
3. **実装（Normal Mode）**: 計画に沿ってコードを書き、テストで検証する
4. **コミット**: 説明的なメッセージでコミットし、PRを作成する

小さな修正（typo、ログ追加、変数名変更）は計画をスキップして直接実装してよい。

### 1.3 具体的なコンテキストを提供する

| 悪い例 | 良い例 |
|--------|--------|
| 「テストを追加して」 | 「foo.pyにログアウト時のエッジケースのテストを書いて。モックは避けて」 |
| 「ログインバグを直して」 | 「セッションタイムアウト後にログインが失敗する。src/auth/のトークンリフレッシュを確認して、再現テストを書いてから修正して」 |
| 「カレンダーウィジェットを追加して」 | 「HotDogWidget.phpのパターンを参考に、月選択とページネーション付きのカレンダーウィジェットを実装して」 |

## 2. コンテキスト管理

### 2.1 コンテキストウィンドウは最重要リソース

Claudeのコンテキストウィンドウは会話全体、読み込んだファイル、コマンド出力で埋まる。パフォーマンスはコンテキストが埋まるほど低下する。

- 無関係なタスク間では `/clear` でリセットする
- 2回以上の修正で解決しない場合は `/clear` して最初からやり直す
- 調査タスクはサブエージェントに委譲してメインコンテキストを保護する
- `/compact` で会話を要約してコンテキストを節約する

### 2.2 セッション管理

- `claude --continue`: 直前のセッションを再開
- `claude --resume`: 過去のセッション一覧から選択
- `/rename`: セッションに名前をつけて管理（例: `oauth-migration`）

## 4. CLAUDE.md の書き方

### 4.1 基本方針

- 200行以内に収める（長すぎるとClaudeが指示を見落とす）
- Claudeが推測できない情報だけを書く
- 具体的で検証可能な指示にする
- 定期的にレビューし、不要な項目を削除する

### 4.2 書くべき内容

- Claudeが推測できないBashコマンド
- デフォルトと異なるコードスタイルルール
- テスト実行方法、推奨テストランナー
- ブランチ命名規則、PR規約
- プロジェクト固有のアーキテクチャ決定
- 環境変数や開発環境の特殊設定
- よくあるハマりポイント

### 4.3 書くべきでない内容

- Claudeがコードを読めばわかること
- 言語の標準的な規約
- 詳細なAPIドキュメント（リンクで代替）
- 頻繁に変更される情報
- ファイルごとの説明
- 「クリーンなコードを書く」のような自明な指示

## 5. チーム開発フロー

### 5.1 ブランチ戦略

```
main ← develop ← feature/xxx
                ← fix/xxx
                ← refactor/xxx
```

### 5.2 コミットメッセージ規約

```
<type>(<scope>): <description>

feat: 新機能
fix: バグ修正
refactor: リファクタリング
test: テスト追加・修正
docs: ドキュメント
chore: ビルド・設定変更
```

### 5.3 PRフロー

1. featureブランチで開発
2. Claude Codeでテスト実行・リント通過を確認
3. PRを作成（テンプレートに従う）
4. 別セッションのClaudeでレビュー（Writer/Reviewerパターン）
5. レビュー指摘を修正
6. マージ

### 5.4 Writer/Reviewer パターン

| セッションA（Writer） | セッションB（Reviewer） |
|----------------------|----------------------|
| 機能を実装 | - |
| - | `@src/path/file.ts のレビューを実施。エッジケース、レースコンディション、既存パターンとの一貫性を確認して` |
| レビューフィードバックを元に修正 | - |

## 6. 禁止事項・注意事項

### 6.1 よくあるアンチパターン

| アンチパターン | 対策 |
|-------------|------|
| キッチンシンクセッション（1セッションで無関係な複数タスク） | タスク間で `/clear` する |
| 何度も修正を繰り返す | 2回失敗したら `/clear` して良いプロンプトで再開 |
| CLAUDE.mdが肥大化 | 不要な指示を削除。コードを読めばわかることは書かない |
| 検証なしで信頼 | テスト、スクリプト、スクリーンショットで必ず検証 |
| 無限探索 | 調査はスコープを絞るか、サブエージェントに委譲 |

### 6.2 セキュリティ

- 機密情報（APIキー、パスワード）をCLAUDE.mdやコミットに含めない
- `.env` ファイルは `.gitignore` に含める
- セキュリティレビュー用サブエージェントを活用する

## 7. スキル・フック・サブエージェントの使い分け

| 機能 | 用途 | 実行タイミング |
|------|------|-------------|
| CLAUDE.md | 常に適用するルール・規約 | 毎セッション開始時に自動読み込み |
| .claude/rules/ | パス別ルール | 対象ファイル操作時に自動読み込み |
| Skills | 再利用可能なワークフロー | ユーザー or Claude が必要時に起動 |
| Hooks | 確実に実行すべき自動処理 | 特定イベント発生時に必ず実行 |
| Subagents | 隔離された専門タスク | Claude が委譲 or ユーザー指示時 |
